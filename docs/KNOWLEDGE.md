# 생기부 개인정보 익명화 도구 - 구현 노하우 및 예외 정책 (v2.0)

## 1. Excel 예외 처리 및 탐색 정책

### A. 병합 셀 (Merged Cells) 처리
- **현상**: openpyxl에서 병합된 셀 중 대표 셀(좌상단 셀)을 제외한 나머지 하위 셀들은 `MergedCell` 타입으로 판정되며 값 조회 및 수정을 시도하면 에러가 발생합니다.
- **정책**: `AnonymizeDetector`에서 스캔 시 `type(cell).__name__ == "MergedCell"` 조건으로 식별하여 스킵합니다. 오직 대표 셀의 텍스트에만 치환 규칙을 매칭하고 저장합니다.

### B. 숨김 시트 (Hidden Sheets) 처리
- **현상**: Excel 통합 문서에 사용자가 숨겨놓은 시트(hidden state)가 존재하여 불필요한 탐색 비용이 발생하거나 개인정보가 누락될 수 있습니다.
- **정책**: `sheet.sheet_state` 가 `'visible'`이 아닌 경우 탐색 루프에서 스킵하여 성능 낭비를 줄이고 사용자 가시 영역만 작업 대상에 포함시킵니다.

### C. 수식 셀 (Formula Cells) 보호
- **현상**: 수식이 기입된 셀(예: `=SUM(...)`)을 일반 문자열처럼 치환하면 수식 자체가 깨지게 됩니다.
- **정책**: 셀의 데이터 값이 `=` 기호로 시작하는 경우 탐색 및 치환 범위에서 배제하여 기존 수식을 보존합니다.

## 2. Safe Save (안전 저장) 4단계 파이프라인

Excel을 직접 저장할 경우 발생할 수 있는 파일 손상 및 비정상 종료 시 데이터 유실을 차단하기 위해 다음 4단계 프로세스를 준수합니다.

1. **1단계**: 원본 파일을 읽어 `temp/` 디렉토리에 고유 식별 명칭(UUID)을 가진 임시 파일 복제본을 만듭니다.
2. **2단계**: openpyxl을 로드하여 서식을 해치지 않고 셀 단위 문자열 치환을 적용합니다.
3. **3단계**: 임시 저장된 Excel 파일을 읽기 모드로 다시 열어 파싱이 제대로 되는지 무결성 검증을 거칩니다 (Open Test).
4. **4단계**: 검증을 성공적으로 통과하면 비로소 사용자가 지정한 output 폴더 경로로 임시 파일을 복사하고 원본과 임시본을 정리합니다. 실패 시에는 덮어쓰지 않고 롤백을 수행합니다.

## 3. 키워드 처리 및 매핑 규칙 (v2.0 업데이트)

### A. 중복 키워드 우선순위 정책 (Deduplication)
- 입력 단계에서 충돌(학생명, 학교명, 삭제어가 겹치는 경우)이 발생하면 아래의 우선순위 사슬에 의해 필터링됩니다.
  $$\text{학생명} > \text{학교명} > \text{삭제어}$$

#### 우선순위 처리 예시 매트릭스
| 원본 입력 | 학생명 목록 | 학교명 목록 | 삭제 단어 목록 | 최종 처리 결과 | 적용 규칙 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **김민수** | 김민수 | 김민수 | 김민수 | **학생1** | 학생명 우선적용 (학교명, 삭제어에서 자동 제외) |
| **서울중** | (없음) | 서울중 | 서울중 | **학교A** | 학교명 우선적용 (삭제어에서 자동 제외) |
| **삭제단어** | (없음) | (없음) | 삭제단어 | **(대체문구)** | 삭제 단어 규칙 적용 |

### B. 인덱싱 결정성 (Determinism)
- Python의 `set`을 사용한 키워드 병합은 해시 시드 난수화에 따라 매번 매핑 순서(학생1, 학생2 등)가 뒤집힙니다.
- 재현 가능한 테스트 및 검수를 위해 중복 제거 시 반드시 순서를 보존하는 `list(dict.fromkeys(...))` 형식을 채택해야 합니다.

### C. 부분 문자열 매칭 오버랩 차단 (Span Overlap Resolution)
- 단순 `in` 연산자 검색 방식 대신, 길이 역순으로 정렬된 패턴들에 대한 문자열 검색 시 이미 치환이 결정된 문자열 스팬(`matched_intervals: list[tuple[int, int]]`)을 기록 추적합니다.
- 새로운 검색 단어가 기존 영역과 오버랩되는 경우(예: '김민수' 매치 완료 후 '민수'가 동일 위치에서 매치 시도될 때) 매칭 대상에서 제외함으로써 부분 매칭 충돌 문제를 알고리즘적으로 원천 해결합니다.

### D. UUID 기반 O(1) 상태 조회 관리
- 테이블 정렬/검색/필터링 기능 확장 시 테이블의 Row Index는 실제 데이터 리스트의 인덱스와 1:1 매치되지 않게 됩니다.
- 따라서 모든 `DetectionItem`에 고유 `item_id`(UUID)를 부여하고 `AppController`에서 `item_id -> DetectionItem` 맵을 유지하여 정렬 상태에서도 O(1) 성능과 데이터 정합성을 보장합니다.

## 4. HWP / HWPX 예외 처리 및 알림 스레드 안전성 정책 (v5.0 추가)

### A. HWPX 직접 XML 치환 정책
- **현상**: HWPX 포맷은 XML 파일들이 zip으로 아카이브된 구조로, 타사 라이브러리 없이 직접 텍스트 노드를 치환하면 속도가 빠르고 서식 훼손 리스크가 없습니다.
- **정책**: Python 표준 `zipfile` 및 `xml.etree.ElementTree`를 활용하여 `Contents/section*.xml` 내의 문단(`<hp:p>`) 하위 텍스트 노드(`<hp:t>`)를 직접 파싱하고 승인된 변경 예정 값만 치환 후 재패키징합니다.

### B. HWP OLE 직접 치환 차단 (PoC 격리)
- **현상**: HWP COM API의 치환 기능(`HAction.Execute("AllReplace")` 등)은 한글의 커서 위치에 의존하므로, 사용자가 검수 테이블에서 개별적으로 선택 승인/해제한 인스턴스의 텍스트 오프셋이 뒤틀려 서식이 파괴되거나 엉뚱한 위치가 치환되는 위험이 큽니다.
- **정책**: HWP 포맷은 `GetText` API를 이용한 **'텍스트 탐색(스캔)'만 제공**하며, 직접 쓰기(치환) 기능은 충분한 한글 행정 서식 테스트 검증이 끝날 때까지 비활성화(`NotImplementedError` 가드)하여 사용자의 원본 파일 서식을 100% 보존합니다.

### C. QSystemTrayIcon 기반 스레드 안전 알림 정책
- **현상**: 백그라운드 Worker 스레드(QThread)에서 동작 완료 시 메인 UI 위젯(QMessageBox 등)을 직접 띄우면 PySide의 GUI 스레드 경계를 침범하여 프로그램 크래시(Crash)가 일어납니다.
- **정책**: `QTimer.singleShot(0, callback)`을 사용해 알림 실행 명령을 메인 GUI 이벤트 루프로 예약 릴레이 송출합니다. 추가 외부 패키지 설치 없이 PySide 내장 `QSystemTrayIcon.showMessage()`를 활용해 백신이나 윈도우 보안 정책 차단 없이 네이티브 알림창을 띄우며, 트레이가 없거나 미지원되는 특수 PC 환경에서는 로그 및 에러 메시지 팝업으로 Fallback 처리합니다.
